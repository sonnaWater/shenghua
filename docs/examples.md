# 各等級壓縮對照集

每組:同一問題在一般回應與四個省話等級下的輸出。

## 1. React 重複渲染

問:「我的 React 元件一直重複渲染,怎麼修?」

- 一般:「這是很常見的問題!元件重複渲染通常是因為每次 render 時都建立了新的物件或函式參照,導致 React 認為 props 有變化。你可以使用 useMemo 或 useCallback 來記憶這些值,這樣就能避免不必要的渲染了。」
- jian:「元件重複渲染,因為每次 render 都建立新的物件參照。用 useMemo 包起來即可。」
- shenghua:「新參照每次 render 都生。useMemo 包住。」
- wenyan:「每繪必生新參照,故重繪。以 useMemo 裹之即止。」
- jiwen:「新 ref→重繪。useMemo 裹之。」

## 2. 資料庫連線池

問:「請解釋資料庫連線池。」

- 一般:「好的!連線池是一種預先建立並維護多條資料庫連線的技術,讓應用程式可以重複使用這些連線,而不需要每次請求都重新建立和關閉連線,這樣可以大幅減少 TCP 握手和認證的成本。」
- jian:「連線池預先建立多條資料庫連線並重複使用,省去每次請求都建立、關閉連線的成本。」
- shenghua:「預建連線,重複用。免每次開關,省 TCP 握手成本。」
- wenyan:「預立連線若干,眾請求共用之。免屢開屢閉,其效大增。」
- jiwen:「預建 conn 池,共用。免開閉之耗。req→借,畢→還。」

## 3. 慢查詢優化

問:「SELECT * FROM orders WHERE YEAR(created_at) = 2025 很慢,兩千萬筆。」

- 一般:「你的查詢慢的主要原因是 YEAR() 函式包住了欄位,這會讓資料庫無法使用 created_at 上的索引,只能做全表掃描。建議改寫成範圍查詢...」
- jian:「YEAR() 包住欄位,索引失效,全表掃描。改範圍查詢:`WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01'`,並確認 created_at 有索引。」
- shenghua:「YEAR() 毀索引→全表掃。改範圍:`WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01'`。補索引。」
- wenyan:「函式裹欄,索引廢,故遍掃全表。改為範圍查詢 `created_at >= '2025-01-01' AND created_at < '2026-01-01'`,索引乃用。」
- jiwen:「YEAR()→索引廢→全掃。範圍查詢 `created_at >= '2025-01-01' AND created_at < '2026-01-01'`。加 index。」

注意:三例中 SQL 皆一字不動 — 程式碼永不壓縮。
